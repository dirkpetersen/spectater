Key Principles

A well-written requirements document allows SpecTater to clearly identify required criteria and provide consistent, accurate pass/fail determinations.

1. Be Specific and Measurable

‚úÖ Good: "Certificate must show General Liability coverage with a minimum limit of $2,000,000 per occurrence"
‚ùå Bad: "Need insurance coverage"

The AI needs concrete values and criteria to compare against. Vague requirements lead to inconsistent evaluations.

2. Use Clear Language

‚úÖ Good: "Certificate holder name and address must match the entity name and primary business address provided in the application"
‚ùå Bad: "Holder info should be correct"

Explicit language helps the AI understand what constitutes compliance.

3. Include Conditions and Exceptions

‚úÖ Good: "Workers' Compensation coverage required for all employees. Exception: Independent contractors with valid 1099 tax status are exempt"
‚ùå Bad: "Need workers' comp"

Context helps the AI apply requirements correctly in edge cases.

4. One Requirement = One Criterion

Break complex requirements into separate items for clarity:

‚úÖ Good:
  • Requirement 1: Certificate must include Certificate Holder name
  • Requirement 2: Certificate must include Certificate Holder address
  • Requirement 3: Certificate Holder name must match entity name from application
‚ùå Bad:
  • Requirement 1: Certificate holder information must be complete and correct

Document Structure

Header Section

Include context at the top to orient the AI:

# [Organization] Insurance Requirements Policy

**Effective Date**: [Date]
**Applicable To**: [Who must comply]

## Overview
Brief description of what this policy covers and why it exists.

Definitions Section (Optional but Recommended)

Define any specialized terms or acronyms:

## Definitions

- **Certificate of Insurance (COI)**: Standard ACORD form 25-S(06/14) documenting active insurance coverage
- **Additional Insured**: Party added to an insurance policy to receive coverage
- **Per Occurrence Limit**: Maximum amount insurer pays per single incident

Requirements Section

Use a numbered list format with clear hierarchy:

## Insurance Requirements

### 1. Certificate of Insurance Information

1.1 Certificate Holder Name
   - **Requirement**: The Certificate Holder name must match the legal entity name
   - **Why**: Ensures the certificate is issued for the correct entity

### 2. General Liability Coverage

2.1 Coverage Type
   - **Requirement**: Certificate must include General Liability (CGL) coverage
   - **Why**: Protects against bodily injury and property damage claims

Practical Examples

Example 1: Certificate Holder Information

Requirement: Certificate Holder name matches the legal entity name from the contract

What passes:
  • Certificate Holder: ACME Contracting LLC
  • Contract Entity: ACME Contracting LLC
  • ‚úÖ PASS: Names match
What fails:
  • Certificate Holder: ACME Contracting, Inc.
  • Contract Entity: ACME Contracting LLC
  • ‚ùå FAIL: Different legal entity type (Inc. vs. LLC)

Example 2: General Liability Limits

Requirement: General Liability per occurrence limit must be at least $2,000,000

What passes:
  • GL Per Occurrence: $2,000,000 ‚úÖ
  • GL Per Occurrence: $5,000,000 ‚úÖ
What fails:
  • GL Per Occurrence: $1,000,000 ‚ùå
  • GL Per Occurrence: Not stated ‚ùå

Example 3: Conditional Requirements

Requirement: Workers' Compensation coverage required for all employees; exempt if contractor has no employees

What passes:
  • Certificate shows: "Workers' Compensation - State Statutory Limits" ‚úÖ
  • Contractor info shows "0 employees" and no WC coverage ‚úÖ
What fails:
  • Certificate shows no WC AND Contractor has 5 employees ‚ùå

Insurance Requirements Checklist

Here are typical insurance requirements you should include:

Certificate Holder / Entity Information

  • Name (must match contract entity)
  • Address (must match registered business address)
  • Business identification

Insurance Company Details

  • Company name
  • AM Best rating (A- VII or better typical minimum)
  • License/authorization to do business in state

General Liability Coverage

  • Per Occurrence limit (e.g., $2,000,000)
  • General Aggregate limit (e.g., $2,000,000)
  • Damage to Rented Premises
  • Additional Insured status

Other Coverages (As Applicable)

  • Umbrella/Excess Liability: If contract value is high
  • Liquor Liability: If serving/selling alcohol
  • Auto Liability: If using vehicles
  • Workers' Compensation: If contractor has employees

Additional Endorsements

  • Waiver of Subrogation
  • Primary and Non-Contributory endorsement
  • Certificate of Insurance issuer contact information

Tips for Success

1. Avoid Ambiguous Language

‚ùå Unclear ‚úÖ Clear
"Reasonable limits" "$2,000,000 per occurrence minimum"
"Appropriate coverage" "$2,000,000 combined single limit"
"Standard protection" "General Aggregate of $2,000,000"

2. Include Dollar Amounts

Don't make the AI guess what's "adequate" or "sufficient." Be explicit with numbers.

3. Document Exceptions Clearly

Be explicit about when requirements don't apply:

‚úÖ "Workers' Comp required for all employees. Exception: Independent contractors (1099 status)"

4. Keep Requirements Focused

Optimal: 5-50 requirements

  • Too few (<5): Minimal value from automation
  • Too many (>100): AI accuracy decreases due to complexity

5. Version Your Requirements

Track changes over time:

Version 1.0 - Initial policy [Date]
Version 1.1 - Added Liquor Liability requirement [Date]
Version 1.2 - Updated General Liability limits [Date]

Using the Template

Option 1: Start with the Template

The repository includes osu-requirements-template.md with insurance requirements pre-filled. You can:

  1. Download and customize the template
  2. Fill in all [TO BE FILLED: ...] placeholders
  3. Upload it to SpecTater to test
  4. Adjust requirements based on results

Option 2: Use Static Requirements

For production deployment, set REQUIREMENTS='osu-requirements-template.md' in your .env file. This:

  • Auto-loads the policy file without user upload
  • Users only upload certificates to evaluate
  • Simplifies the user interface

Option 3: Create from Scratch

Use the structure and examples above to create your own requirements document:

  1. Start with a header section
  2. Add definitions for specialized terms
  3. List requirements with specific criteria
  4. Include examples of pass/fail cases
  5. Test with sample documents

Testing Your Requirements

After writing your requirements:

  1. Upload a known-good certificate
  2. Verify results are accurate
  3. If results are wrong, check requirement clarity
  4. Add more specific language if needed
  5. Repeat until satisfied

Troubleshooting

If SpecTater results are inconsistent or wrong:

  • Check Specificity: Add exact dollar amounts, dates, company names
  • Check Clarity: Avoid words like "reasonable," "adequate," "substantial"
  • Check Completeness: Add examples of what PASS and FAIL look like
  • Enable Debug Mode: Set FLASK_DEBUG=True in .env to see full JSON response
